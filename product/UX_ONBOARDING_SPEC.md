# UX Onboarding Spec

I2P Connect onboarding should make private communication over I2P understandable without fear or hype.

## Experience Goals

Users should feel:

- oriented
- calm
- in control
- aware of limits
- able to take the next safe step

## Onboarding Principles

- Start with current mode and honest limits.
- Explain what data stays local.
- Explain what is measured.
- Explain what is experimental.
- Give one next action at a time.
- Avoid jargon unless it is defined nearby.

## Mission Sequence

1. Understand I2P Connect.
2. Check local I2P readiness.
3. Create a local identity.
4. Verify a contact invite.
5. Send a first private message after implementation and tests exist.
6. Try an experimental audio note after messaging foundations exist.

## Copy Rules

Use measured wording:

- "local status"
- "router reachable"
- "SAM session ready"
- "message queued"
- "experimental"
- "not externally audited"

Avoid unsupported wording:

- guaranteed anonymity is not promised
- full E2EE is not promised until implemented and tested
- real-time video is not promised
- SASE and zero-trust compliance are not promised

## Risk Moments

Future UI must show extra guidance before:

- creating identity material
- exporting identity material
- sharing invites
- importing contacts
- enabling logs
- exporting diagnostics
- recording audio
- enabling optional integrations
